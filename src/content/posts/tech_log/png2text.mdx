## The Story：手写笔记引发的故事

最近在学量子力学，比较常用的本子是拍纸本，是那种没办法整本保存，想用下一张只能把上一张撕下来的本子。这确实很适合随堂笔记的情景，它足够灵活，本质上等同于活页本，只是你需要自己装订。

不过呢，每次回到家的时候，我都会把当天的笔记纸随手往桌上一扔，然后就去做别的事情了。时间一长，笔记难免打堆，而且因为是散页，也很容易丢失。

于是就萌生了一个把手写笔记转化为电子笔记的想法。我做手写的纸质笔记，纯粹是因为我对 handwriting 的热爱，但实在是太难以保存了。所以我想要兼顾二者的优点，能够在电脑上比较好的查看，同时又能满足自己对书写的 passion。

最初想到是用 obsidian 来整理笔记，于是很自然想到看看 obsidian 有没有什么插件可以用。发现并没有符合自己要求的插件，就打算自己写一个。

具体来说，想把手写笔记转换成可用的 markdown 笔记有几个问题，一是公式和批注的界限不够清晰，二是手写笔记中很难避免多列排版，以及跳跃式对某个内容进行补充的部分。如果只是直接按照原来的排版转换，必然会使得整个 markdown 变得不可读，所以一定是需要一种重排版的机制的。

因此就想到可以通过 OCR 识别行间公式和行内公式，将它们分成不同的 block，然后在 obsidian 上做一个前端，可以拖动这些 block 来排版。

测了一下几个 sota 级别的 OCR 模型效果都不好，一是对于手写公式的识别率不佳，二是对于混杂了公式和汉字的场景，会错误的识别行间公式块 / 行内公式块，三是 OCR 模型在这种手写笔记的场景下极大概率出现锁死在某一组循环 token 的情况，导致事实意义上不可用，四是即使区分了公式块，但由于前三点，导致重新排版的时间成本甚至远高于对着笔记自己再手动写一份。这种场景对于 OCR 开始明显有些过于压力测试了，我意识到虽然 OCR 声称它能支持手写公式，但一旦把公式和文字混杂在一起，识别的难度就高出了好几个数量级，这或许也是 more is different 的一种体现吧。

那把 OCR 模型换成 llm 是不是就解决了呢？通过 workflow 编排的方式，第一个 llm 负责 convert，然后 runtime 把第一个 llm 输出的结果给第二个 llm，第二个 llm 再进行公式分块？这就是我写的第二版。

很遗憾，它显然是不行的。由图片转换到文本的过程中丢失了信息，导致第二个 llm 完全不知道原来的排版是怎么样的，它只能根据一个极可能会有问题的 markdown 去划分公式块，效果肯定是不好的，事实证明它只解决了问题一和问题三。

那让同一个 llm 做这件事情怎么样呢？如果只是让它简单的 convert，效果依然是不好的。昨天我突然意识到了这点，于是写了一个很简单的 prompt 丢给 gemini web chat 想看看效果。

> 你可以帮我识别一下这个笔迹，并且按照内容的意图重新排版成可以发表成 blog 的 markdown 文档吗？

测试内容分别是 [傅里叶变换](https://stardust-mono.vercel.app/posts/math_and_physics/fft) 和 [量子力学中的一维谐振子](https://stardust-mono.vercel.app/posts/math_and_physics/infinite_well_and_harmonic_oscillator)，可以看到效果是不错的，并且它的识别准确率居然达到了 99.9%。

这让我意识到，或许之前我们走入了一些思维定势，于是对这个问题有了一些思考，并写了这篇文章。

## Insight：不是图像识别，而是"逻辑填空题"

回过头来看，最初尝试的几个方案之所以失败，其实都源于一个根深蒂固的误区：我把这个任务纯粹当成了一个"视觉提取"或者"图像识别"（Perception）的问题。

传统 OCR 的底层逻辑是确定性的。它试图在二维的纸面上画框，把那一坨模糊的墨迹强行映射成某个字符。但写物理推导的时候，在课堂需要快速记录的场景下，书写的整洁度比较难以得到保障。经常需要在算符之间龙飞凤舞地连笔（并且我还不会写 $\xi$），传统模型看到这种东西只会觉得这是视觉噪声，甚至干脆死循环。

但当大模型（LLM）介入并且具备了多模态能力后，它去"看"这张图的方式，本质上是在做 Next Token Prediction。它像是一个学过量子力学的助教，哪怕视觉编码器对某个糊成一团的偏导符号 $\frac{\partial}{\partial t}$ 置信度极低，它也能根据上下文的物理定律、等式两边的量纲，直接在概率空间里把正确的符号给填进去。这并不是图像识别，而是在做带有视觉提示的逻辑推理。

## The Missing Context：让大模型自己决策

这就引出了那个真正致命的问题：排版与逻辑的割裂。

我在写第二版 workflow 时，试图用第一个 LLM 提取纯文本，再让第二个 LLM 去排版，结果惨败。因为在第一步从"2D 图像"坍缩成"1D 文本流"的过程中，那些跳跃式的箭头、旁边的随手批注、公式之间的空间关系——也就是我推导时的"思维上下文"——彻底丢失了。后一个模型拿到的是一堆完全失去结构的文本（甚至可以说像是被二向箔拍扁了），自然无从下手。

更重要的是，为了追求速记，在纸上留下的往往只有核心逻辑（公式），而去掉了描述性语言和逻辑

所以，真正的解法不是去「识别」然后「切割」，而是把「是否需要补充文字以连贯逻辑」的决策权，直接交给那个看到全貌的大模型。它不需要像强迫症一样完美复刻笔记的物理排版，它需要做的是语义重构（Semantic Reconstruction）。让它去推理这张草稿纸上的思维跳跃，用自然语言把干巴巴的核心逻辑重新润色、串联成一篇可读的 Markdown。人退居二线，只做最后一步的审计（Audit）。事实证明，这不仅治好了排版强迫症，效果也远超死板的串联管线。

## The Giant Gap：从死板的 API 到自主的 Agent

从把 LLM 当成高级 OCR 工具，到把它当作一个能理解上下文、能自主补充逻辑的「重构引擎」，这中间其实跨越了一道认知的鸿沟。

我们太习惯经典软件工程那种「输入 A 必然得到 B」的绝对确定性的映射了，以至于面对一个具备庞大常识库和推理能力的引擎时，第一反应往往是削足适履，把它硬生生塞进流水线的某个节点里当一个单纯的转换器（Converter）。

但面对高度非结构化、充满模糊意图的手稿，最好的方式是完成一次控制权的让渡。跳出单线 workflow 的思维，把它看作一个能够自主处理复杂信息的节点。

当我们在构思更底层的 AI 运行时或架构时，这也提供了一个绝佳的视角：不该再用僵化的数据流去限制模型，而是应该赋予它足够的自主权。把混乱的现实状态（一页手稿）丢给它，它通过系统调用般的内部消化，还你一个结构化、干净的逻辑态。

这或许才是我们从单纯的 png2text 方案中，真正应该看到的未来。
