最近在一个群里，因为对 Agent 工具链的设计分歧，发生了一段挺有意思的讨论。

起因是有人觉得「完全抛弃 MCP（Model Context Protocol）转向 CLI + Skill 是不对的」。其实我并不反对 MCP 本身，但我发现，大家在反驳 CLI 的时候，往往并没有反驳到点子上。

很多人出于直觉，觉得有些东西（比如 IDA、dnSpy 这种带界面的逆向工具）如果 CLI 化了，会让人觉得「非常难受」。这其实暴露了目前我们在构建 Agent 系统时，一个极其典型且根深蒂固的认知谬误。

于是对这个问题有了一些思考，写了这篇文章。

## 认知的十字路口：什么是真正的 AI-Native？

当你觉得把一个图形化软件变成纯命令行会「很难受」时，你其实是在用碳基生物的视觉处理极限，去揣测硅基生物的算力。

人类逆向工程师确实需要 IDA 的 GUI，需要看控制流图、看高亮、看跳转箭头。如果没有图形界面，人确实会疯掉。但大模型需要吗？它不需要。对于大模型来说，你给它一个纯文本的 GDB 内存快照，或者一堆标准输出的寄存器状态，它一样能光速在庞大的上下文窗口里建立起逻辑拓扑。

这就引出了一个非常核心的认知分叉路口：对人来说 native 且好用的东西，对 LLM 来说并不 native。

「我觉得没法用，所以 LLM 也没法用」的这种直觉，本质上是一种拟人化的迷思。真实情况恰恰与这种迷思相悖，CLI 就是对的，AI 对 bash 管道流天生就是 native 的，而且作为一个庞大的参数矩阵，它根本不怕复杂的命令行参数。反而是我们在 GUI 时代习惯的那些交互逻辑，对 AI 来说既冗余又低效。

## MCP 的本质与不可承受之重

很多人把 MCP 当成某种接入外部 Agent 生态的魔法，但我们如果剥开表象，MCP 的本质其实非常简单：它就是一个 tool calling 的配套设施，或者说，它是一种结构化的 RPC（Remote Procedure Call）协议。
既然是 RPC，那它为了实现「远程连接」和「跨进程通信」，就必然付出了极大的性能与架构代价。它的协议栈太重了。

如果我们不需要 tool calling 严格的结构化约束，那我们根本就不需要 MCP。更何况，通过 CLI 加上 Skill 的分层披露，完全可以避免把全量工具的 JSON Schema 一股脑塞进 Prompt 里，直接就把 MCP 动辄撑爆上下文的问题给降维解决了。

有人说 MCP 的优势是即插即用。但在一个纯粹的隔离沙箱里，是费劲去配置节点、连一个 MCP Server 快，还是直接 wget 下载一个二进制 CLI 丢进环境里直接跑快？答案显而易见。

## Stateful 的诅咒与分发的未来

这就触及到了另一个更底层的问题：为什么过去的系统设计会如此依赖 RPC（比如现在的 MCP），而不是直接分发工具本身？

答案是因为过去的 tool 不好分发。传统的工具往往是 stateful（有状态）的。它们深深耦合了宿主机的文件系统、数据库连接池或者特定的内存环境。即使你直接把代码分发给 Agent 的沙箱，你也拿不到服务器内部的那些状态，工具仍然是跑不起来的。

但随着我们构建出真正无状态（stateless）的 Agent，并通过 VFS（虚拟文件系统）接管了状态管理，这个历史包袱就不存在了。既然工具可以是无状态的逻辑单元，直接分发 CLI 执行文件显然比拖着一个沉重的 RPC 协议要优雅得多。

## Syscall vs Toolkit：最终的架构和解

但这是否意味着 MCP 毫无用处？并不是。

在群里讨论的最后，我给出了我的定论：CLI 可以完全覆盖所有场景，但我们仍然会需要 Built-in tool，并通过 tool calling 来作为 syscall，所以 MCP 并没有被 deprecated。

这里的核心在于边界的划分：
CLI 不是某一个特定的外部 tool，它是一个泛用的执行环境（Toolkit）。它是跑在用户态的应用程序，负责处理海量的、需要灵活组合的外部任务。

而 tool calling 及其背后的 MCP 协议，它的结构化语义在一种情况下是绝对必要的——那就是当 Agent 需要调用底层宿主机的核心原语（Built-in tool）时。这时候，MCP 就扮演了操作系统中系统调用（Syscall）的角色。因为在跨越内核边界时，我们需要极其严格的结构化校验，以降低 LLM 直接拼接复杂转义字符所带来的认知成本。

就像我在群里最后说的那样：“结论一样，不代表我们的想法是一样的。”

面对复杂的 Agent 架构，只有剥离掉直觉上的喜恶，用第一性原理去审视每一行数据流的开销，才能真正明白我们到底在造一个什么样的系统。
